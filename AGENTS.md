# AGENTS.md

## Scope

This file applies to the whole `ankhorage/studio` repository.

`@ankhorage/studio` is the standalone Studio authoring package for Ankhorage apps. Keep it useful as a package-neutral authoring/model package. Do not couple it to the `ankhorage4` transition host, generated app internals, concrete Zora implementation modules, Supabase storage, Expo Router UI, or a specific runtime adapter.

## Repository facts

- Package name: `@ankhorage/studio`.
- Runtime: Bun.
- Language: TypeScript, ESM, strict mode.
- Main source root: `src/`.
- Build output: `dist/`.
- Public root export: `src/index.ts`.
- Manifest-state subpath: `src/manifestState.ts` exported as `@ankhorage/studio/manifestState`.
- README is generated by Paradox. Do not edit `README.md` manually; update source/docs and regenerate where applicable.

## Core architectural rules

### Studio owns authoring/product contracts and package-neutral model helpers

Studio owns reusable authoring contracts, command/event contracts, package-neutral authoring model helpers, and package-neutral manifest-state helpers.

Good:

- Studio selection/session/context contracts
- action registry contracts
- insert catalog model helpers
- placement/tree helpers
- manifest state mutations that depend only on contracts and caller-provided metadata
- docs explaining Studio package boundaries

Avoid:

- React hook lifecycle/state effects in the package-neutral model
- React Native UI components
- Expo Router or navigation UI implementation
- DnD runtime implementation
- Supabase/storage implementation
- generated-app runtime overlay implementation
- concrete Zora imports
- template catalog ownership

### Component metadata is injected

Do not import concrete Zora metadata into this package. When placement or catalog logic needs component rules, accept `StudioComponentMetaRegistry` from the caller.

This keeps `@ankhorage/studio` usable by hosts other than the current `ankhorage4` transition host.

### Keep runtime boundaries clear

Do not move generic runtime rendering behavior into Studio. Runtime rendering belongs to `@ankhorage/runtime` and platform/runtime planning belongs to packages such as `@ankhorage/expo-runtime`.

Studio may define authoring contracts that refer to manifests, but it should not own generated app runtime implementation.

## Public API rules

Public exports are explicit.

When adding a public helper or type:

1. export it from the intended public entrypoint
2. add tests that import the public symbol so Knip recognizes the public surface
3. document the package boundary when the new helper changes ownership
4. add a changeset for published API changes

Prefer the root export for broad contracts and authoring helpers. Prefer `@ankhorage/studio/manifestState` for manifest-state model helpers so downstream hosts can migrate incrementally.

Do not rely on deep imports from consumers. If consumers need it, export it intentionally.

## Manifest-state rules

`src/manifestState.ts` must stay package-neutral.

Allowed:

- pure manifest fingerprinting and sync comparison helpers
- active screen/root/selection resolution
- route tree helpers
- pure node/screen/theme/data/module/OAuth manifest mutations
- helpers that accept `StudioComponentMetaRegistry` from callers

Not allowed:

- React `useState`, `useEffect`, `useMemo`, or hook lifecycle code
- UI panels, previews, canvas components, or admin screens
- concrete Zora imports
- Supabase/storage calls
- generated-app runtime overlay code

## Tests and validation

Run the relevant checks before handing off.

Preferred full validation:

```bash
bun run build
bun run lint:fix
bun run test
bun run knip
bun run typecheck
bun run format:check
bun run docs
bun run changeset:status
```

Available package scripts:

```bash
bun run build
bun run changeset
bun run changeset:status
bun run docs
bun run knip
bun run lint
bun run lint:fix
bun run format
bun run format:check
bun run test
bun run typecheck
bun run version-packages
```

Notes:

- `bun run test` runs Bun tests.
- `bun run typecheck` runs TypeScript with `--noEmit`.
- `bun run build` emits package output into `dist`.
- `bun run docs` runs Paradox.
- `README.md` is generated by Paradox; do not edit it manually.

## Changesets

Add a changeset for any public-facing change:

- new public export
- exported type or helper change
- package export/subpath change
- manifest-state behavior change
- dependency or public package shape change

Use patch unless the change is clearly minor/major.

No changeset is usually needed for:

- internal-only refactor
- test-only changes
- docs-only changes that do not affect published package content
- non-public implementation cleanup

When unsure, add a patch changeset.

## Dependency rules

Current direct dependency:

- `@ankhorage/contracts`

Do not add large runtime dependencies casually. For new dependencies:

- justify why Studio needs it directly
- prefer caller injection for host/platform/runtime concerns
- avoid bringing generated-app/runtime/product host concerns into this package

## Working style for agents

Before coding:

1. inspect current public exports
2. inspect related tests and docs
3. identify the owning package boundary
4. identify whether a changeset is needed

While coding:

1. keep changes narrow
2. preserve public API compatibility where possible
3. avoid host/runtime/platform coupling
4. update tests with behavior
5. update docs/changeset for public package changes

Before final handoff:

1. run validation or clearly report what was not run
2. include the PR/commit reference
3. mention public API changes
4. mention changeset path
5. mention any follow-up issue needed

## Common pitfalls

- Editing generated `README.md` manually.
- Importing Zora into package-neutral Studio helpers.
- Moving React hook lifecycle into the manifest-state model.
- Adding Supabase, Expo Router, DnD, or generated-app runtime implementation code.
- Adding public exports without tests, causing Knip failures.
- Forgetting a changeset for public package changes.
- Treating `ankhorage4` transition host behavior as package-owned behavior.

## Current architectural direction

Keep this separation:

```txt
Studio
  authoring contracts, product contracts, package-neutral authoring/model helpers

ZORA
  app-facing UI + static component metadata

Runtime
  manifest rendering, registry, action/data binding

Platform runtime packages
  Expo/native adapters, permission/capability planning, config plugins

ankhorage4
  transition/integration host consuming extracted packages

Generated apps
  output consumers of manifests, runtime, ZORA, and platform packages
```
